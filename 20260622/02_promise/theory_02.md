# Promise: 실무 중심의 비동기 흐름 제어

## 학습 목표
*   Promise의 기본 정의와 3가지 내부 상태(Pending, Fulfilled, Rejected)를 Geolocation API 실무 사례를 통해 학습합니다.
*   .then(), .catch(), .finally() 메서드의 유기적 결합을 프로필 업로드 로딩 제어 실무 코드를 바탕으로 구현합니다.
*   쇼핑몰 결제 주문 파이프라인을 구축하여 콜백 지옥을 극복하고 에러 처리를 일괄 단일화하는 Promise 체이닝을 마스터합니다.
*   독립된 대시보드 위젯 API 데이터들을 Promise.all 병렬 처리로 최적화하여 3배 빠른 렌더링 성능 향상법을 비교 분석합니다.

## 1. Promise의 개념과 3가지 상태 (실무 사례: 사용자 위치 권한 요청)

### 1) Promise란 무엇인가
*   비동기 연산의 최종 완료 또는 실패와 그 결과값을 나타내는 객체입니다.
*   자바스크립트에서 즉시 값을 반환할 수 없는 통신이나 디바이스 제어 명령을 내릴 때, 미래의 어떤 시점에 성공 데이터 혹은 실패 에러를 반환하겠다는 약속을 추상화한 표준 객체입니다.

### 2) Promise의 3가지 상태 (State)
브라우저의 실무 API인 **Geolocation API (사용자 위치 정보 획득)** 를 호출할 때, 브라우저는 내부적으로 Promise의 3가지 상태 전이를 그대로 수행합니다.

*   **Pending (대기)**: 비동기 작업이 아직 완료되지 않았거나 대기 중인 상태입니다.
    *   **Geolocation 대입**: 브라우저 상단에 "위치 권한을 허용하시겠습니까?" 팝업창이 뜨고, 사용자가 선택을 하기 전까지 머물러 있는 상태입니다.
*   **Fulfilled (이행 완료)**: 비동기 작업이 성공적으로 완료되어 결과 데이터가 반환된 상태입니다.
    *   **Geolocation 대입**: 사용자가 "허용"을 클릭하여 브라우저가 현재 GPS 위도/경도 정보를 성공적으로 수집하여 값을 반환한 상태입니다. (resolve 호출)
*   **Rejected (거부 완료)**: 에러가 발생하여 비동기 작업이 실패하고 오류를 반환한 상태입니다.
    *   **Geolocation 대입**: 사용자가 "차단"을 클릭하여 권한을 거부했거나, 장치의 GPS 신호 감지에 실패한 상태입니다. (reject 호출)

### 3) 상태 전이 아스키 다이어그램
```text
               +----------------------------------------+
               |           Pending (대기 상태)           |
               |     (예: 위치 권한 팝업 수락 대기)       |
               +----------------------------------------+
                  /                                  \
     사용자 권한 승인 (resolve)             사용자 권한 거부 (reject)
                /                                      \
               v                                        v
+-------------------------------+        +-------------------------------+
|     Fulfilled (이행 완료)      |        |      Rejected (거부 완료)      |
|    (GPS 위도/경도 좌표 반환)    |        |     (권한 거부 에러 메시지)     |
+-------------------------------+        +-------------------------------+
```

## 2. .then(), .catch(), .finally() 흐름 제어 (실무 사례: 프로필 업로드 및 ID 실시간 검증)

### 1) 실무 상황에서의 리소스 정리 및 실시간 유효성 검증
*   비동기 통신을 시도할 때 사용자에게 진행 상태를 알리기 위해 로딩 스피너를 작동시키거나, 입력창에 아이디를 칠 때 실시간으로 중복 여부를 백엔드와 맞추어 확인하는 유효성 체크가 필요합니다.
*   2일차에 배운 **`input` 이벤트(e.target.value)** 와 **Promise** 를 결합하여, 사용자가 타이핑하는 동안 비동기적으로 중복 상태를 조회하고 성공 상태와 오류 상태를 즉각 UI에 하이라이트합니다.
*   이때 가장 중요한 점은 **성공하든 실패하든 무조건 화면의 로딩 상태를 해제하고 다음 인터랙션을 대기해야 한다**는 것이며, 이를 보장하는 통로가 바로 `.finally()`입니다.

### 2) 각 메서드의 흐름 분석
*   **.then(callback)**: Promise가 Fulfilled 상태가 되었을 때 실행되며, 중복 검증 성공 메시지를 띄우거나 업로드 완료 이미지 파일 URL을 화면에 바인딩합니다.
*   **.catch(callback)**: 아이디 길이가 짧거나 형식이 맞지 않는 경우, 또는 이미지 크기가 5MB 초과하여 Rejected 상태가 된 경우 실행되어 직관적인 경고 문구를 UI에 반영합니다.
*   **.finally(callback)**: 통신의 성공/실패 유무와 관계없이, 비동기 작업 파이프라인의 최종 마감재로서 스피너를 끄고 로딩 상태를 정돈합니다.

## 3. Promise 체이닝을 통한 순차 제어 (실무 사례: 쇼핑몰 주문 결제 파이프라인)

### 1) 왜 순차 비동기 체이닝이 필요한가
*   현업에서 비동기 작업은 앞선 작업의 결과물을 입력값으로 받아 동작하는 경우가 많습니다.
*   예를 들어, 상품 주문을 위해서는 `1단계: 주문서 생성`을 통해 고유한 `orderId`를 발급받아야 하고, 그 `orderId`를 사용하여 `2단계: PG사 결제 요청`을 보낸 뒤 `paymentId`를 획득해야 하며, 마지막으로 `3단계: 영수증 알림톡 발송 API`에 결제 ID를 전달해야 합니다.

### 2) 콜백 지옥과 Promise 체이닝의 에러 처리 차이점
*   **콜백 방식**: 각 단계마다 오류 처리를 수작업으로 감싸야 하며, 중첩이 누적되어 코드가 내밀립니다.
*   **Promise 체이닝**: 각 `.then()`의 콜백 내부에서 새로운 Promise를 반환(`return`)함으로써 코드를 평평하게 일렬로 유지합니다.
*   특히 **에러 전파(Error Bubbling)** 속성 덕분에, 1단계(주문), 2단계(결제), 3단계(알림) 중 **어느 곳에서 에러가 발생해도 중단 즉시 하단의 단 하나의 `.catch()`로 모든 예외 처리가 통합 수집**됩니다.

```javascript
// 쇼핑몰 주문 결제 API 연쇄 체이닝 형태
createOrder(cartItems)
  .then(orderId => processPayment(orderId))
  .then(paymentId => sendReceiptNotification(paymentId))
  .then(resultText => {
    // 모든 연쇄 성공 시 최종 UI 처리
    console.log(resultText);
  })
  .catch(error => {
    // 1, 2, 3단계 중 어느 하나의 예외라도 이쪽으로 즉시 점프하여 대응
    console.error("주문 처리 실패 안내:", error.message);
  });
```

## 4. Promise.all을 활용한 병렬 비동기 최적화 (실무 사례: 대시보드 화면 로딩)

### 1) 대시보드 로딩 시 순차와 병렬의 속도 차이
*   사용자가 관리자 대시보드에 접근했을 때, 화면 구성을 위해 `사용자 프로필 데이터`, `실시간 매출 통계 차트`, `배너 광고 및 공지사항` 세 가지 데이터를 수신해야 합니다.
*   이 세 가지 데이터는 상호 의존성이 없으므로 굳이 순서대로 기다릴 필요가 없습니다.

### 2) 타임라인 비교 아스키 아트
*   **순차 요청 (비효율적 패턴)**: 전체 소요 시간은 각 대기 시간의 합산이 되어 화면 로딩이 극도로 정체됩니다.
```text
[순차] 프로필 조회(0.8초) ======> 통계 데이터(1.2초) ========> 공지 조회(0.6초) ======> 총 2.6초 대기
```
*   **병렬 요청 (Promise.all 적용)**: 3가지 비동기 요청을 동시에 출발시켜, 전체 로딩 시간이 세 가지 중 가장 오래 걸리는 작업 시간(1.2초)으로 묶입니다.
```text
[병렬] 프로필 조회(0.8초) =======>
       통계 데이터(1.2초) ==========================> (총 1.2초 대기 후 일괄 화면 갱신 완료)
       공지 조회(0.6초)   ====>
```

### 3) Promise.all의 특징과 주의점
*   `Promise.all([p1, p2, p3])`은 전달된 배열 속 모든 Promise가 성공할 때만 최종 성공 배열을 반환합니다.
*   전달된 Promise 중 **단 하나라도 실패(Rejected)하면 다른 성공 여부와 무관하게 즉각 전체 실패 처리가 구동** 되어 하단 catch 문이 호출되므로, 실무 적용 시 개별 API 통신 실패에 대한 대비책을 염두에 두어야 합니다.
