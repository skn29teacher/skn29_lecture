#  동기 vs 비동기, 그리고 AJAX의 역사

## 학습 목표
*   동기와 비동기 처리의 근본적인 차이점을 실생활의 직관적인 비유를 통해 이해합니다.
*   자바스크립트가 싱글 스레드임에도 불구하고 비동기 작업이 가능한 원리인 이벤트 루프(Event Loop)와 브라우저 아키텍처를 분석합니다.
*   현대 웹 애플리케이션의 핵심 기틀이 된 AJAX 및 XMLHttpRequest(XHR)의 역사적 배경을 파악합니다.
*   비동기 처리를 조작하기 위해 등장했던 초기 콜백 패턴의 동작 방식과 콜백 지옥(Callback Hell) 문제를 인식합니다.

---

## 1. 동기(Synchronous) vs 비동기(Asynchronous)

### 1) 개념 비교
*   **동기(Synchronous)**: 작업이 순차적으로 실행되며, 하나의 작업이 완전히 끝날 때까지 다음 작업은 대기(Blocking)해야 합니다.
*   **비동기(Asynchronous)**: 작업을 지시한 후 완료를 기다리지 않고 바로 다음 코드로 제어권이 넘어가는 비차단(Non-blocking) 방식입니다.

### 2) 복습: 이벤트 버블링(Propagation)과 전파 방지
*   동기/비동기 블로킹 확인을 위해 반응성 테스트 기기(버튼)를 클릭할 때, 자바스크립트의 **이벤트 전파(Event Propagation) 버블링** 현상이 유기적으로 관측됩니다.
*   버튼 클릭 시 이벤트가 부모 카드(`.section-card`) 컨테이너로 버블링 상승하는 현상을 콘솔 로그와 터미널을 통해 추적합니다.
*   `e.stopPropagation()`을 실행했을 때, 이벤트 버블링 전파가 완벽히 차단되어 부모 카드가 이벤트를 감지하지 못하는 2일차 핵심 메커니즘을 다시 한번 복습합니다.

### 3) 실습 내용
*   동기식 강제 대기 함수를 호출하면 자바스크립트 단일 스레드가 완전히 차단되어, 그 시간 동안 화면의 다크모드 버튼 클릭이나 텍스트창 입력 등의 인터랙션이 정지(블로킹)되는 현상을 직접 확인합니다.
*   버블링 제어 스위치(체크박스)를 켜고 끔에 따라 클릭 이벤트의 전파 양상이 어떻게 달라지는지 검증합니다.

## 2. 자바스크립트 엔진과 브라우저 이벤트 루프 (Event Loop)

### 1) 실무적인 관점에서의 이벤트 루프
*   동기식 무거운 연산(예: 수천만 건의 데이터를 가공하고 화면에 그리는 일)은 호출 스택(Call Stack)을 오랫동안 점유하여 브라우저 화면을 얼어붙게 만듭니다.
*   이러한 대규모 작업을 잘게 쪼개어(Chunking), 한 조각이 끝날 때마다 `setTimeout(..., 0)`을 통해 **다음 연산 조각을 이벤트 루프의 Task Queue 뒤로 위임** 하는 기법은 현대 웹 성능 최적화의 핵심입니다.
*   자바스크립트 엔진이 한 조각을 연산한 직후 브라우저에 스레드 제어권을 양보하여 화면을 갱신(로딩 게이지 증가 등)하고 사용자 이벤트를 처리한 뒤, 큐에서 다음 연산 조각을 꺼내 실행하는 구조가 이벤트 루프의 대표적인 실무 활용 사례입니다.

### 2) 아스키 다이어그램
```text
  [ Call Stack ]               [ Web APIs ]
+----------------+          +--------------------+
|                |  요청    | setTimeout/DOM/XHR |
|   현재 함수    | -------> |                    |
+----------------+          +--------------------+
        ^                             |
        |                             | 완료 시 콜백 등록
   이벤트 루프                        v
  호출스택 감시            [ Task Queue ]
  (비어 있으면)            +--------------------+
        |<---------------- | 대기 중인 콜백들    |
                           +--------------------+
```

### 3) 실습 내용
*   3천만 번의 복잡한 수학 연산을 수행할 때, 동기식 일괄 처리와 이벤트 루프 분할 처리의 성능 격차를 확인합니다.
*   동기식 처리 시에는 연산 도중 로딩바가 0%에서 완전히 멈춰 있다가 작업이 끝나는 순간 100%로 순간이동하지만, 이벤트 루프 분할 처리 시에는 0%에서 100%까지 게이지바가 실시간으로 부드럽게 올라가며 화면 테마 클릭도 원활히 동작하는 렌더링 혜택을 확인합니다.

## 3. AJAX의 탄생과 XMLHttpRequest (XHR)

### 1) AJAX의 의의
*   페이지 전체를 새로 불러오지 않고, 백그라운드 통신을 거쳐 JSON/XML 데이터를 수신한 후 DOM API를 이용해 화면 일부만 동적으로 조작하는 현대 웹 인터랙션의 시초입니다.
*   브라우저 표준 도구인 XMLHttpRequest (XHR) 객체를 기반으로 동작합니다.

### 2) XHR readyState 라이프사이클 (우편 배달 비유)
서버와 비동기 통신을 시도할 때, 브라우저는 내부 상태를 `readyState`라는 숫자로 나타냅니다. 이것은 편지를 주고받는 배송 과정에 빗대어 쉽게 이해할 수 있습니다.

*   `0 (UNSENT) - 편지 작성 준비 단계`
    *   **설명**: XHR 객체(`new XMLHttpRequest()`)만 겨우 생성되고, 아무런 연결 지시를 하지 않은 빈 껍데기 상태입니다.
    *   **비유**: 우체국에 가기 전, 편지지에 내용을 쓰고 봉투에 넣기만 한 상태입니다.
*   `1 (OPENED) - 우체국 접수 단계`
    *   **설명**: `open()` 메서드가 호출되어, 어떤 방식으로(GET/POST) 어느 주소(URL)로 보낼 것인지 통신 설정이 완료된 상태입니다.
    *   **비유**: 우체국 창구에 편지를 접수하여 보낼 준비가 완료된 상태입니다.
*   `2 (HEADERS_RECEIVED) - 상대방 도착 및 영수증 회신 단계`
    *   **설명**: `send()`를 통해 실제 요청을 보냈고, 서버로부터 응답 헤더(성공/실패 여부를 알려주는 메타데이터와 상태코드)를 가장 먼저 수신한 상태입니다.
    *   **비유**: 편지가 수신인의 집 우편함에 정상적으로 도달하여 배달 완료 알림 영수증을 우체국으로부터 받은 상태입니다.
*   `3 (LOADING) - 짐 내리기 단계`
    *   **설명**: 서버로부터 수신할 실제 본문 데이터(HTML이나 JSON 등)를 패킷 단위로 전송받고 있는 다운로드 진행 중 상태입니다.
    *   **비유**: 소포 내용물(실제 데이터)을 우편함에서 거실 테이블 위로 하나씩 무거운 짐을 풀어서 나르고 있는 상태입니다.
*   `4 (DONE) - 편지 개봉 및 읽기 완료 단계`
    *   **설명**: 모든 데이터를 에러 없이 온전하게 다운로드 완료하여 사용할 준비가 최종 끝난 상태입니다.
    *   **비유**: 거실 테이블 위에 편지를 전부 펼쳐놓고 내용을 소리 내어 다 읽은 완성 상태입니다. 이때부터 자바스크립트로 데이터를 화면에 뿌릴 수 있습니다.

### 3) XHR의 비동기적 흐름과 실행 순서의 증명
XHR 코드가 작성된 한 줄 한 줄은 위에서 아래로 순차 실행되지만, 통신 행위 자체는 완전한 비동기로 구동됩니다. 이를 코드 호출 스택 관점에서 증명해 봅니다.

*   **예시 코드 흐름**:
    1.  `const xhr = new XMLHttpRequest();` (객체 생성 - 즉시 실행)
    2.  `xhr.open("GET", "https://...", true);` (설정 정보 셋업 - 즉시 실행)
    3.  `xhr.onreadystatechange = () => { ... };` (상태 감시 이벤트 리스너 등록 - 즉시 완료)
    4.  `xhr.send();` (요청 신호 발송 - 발송 즉시 자바스크립트 스레드는 제어권을 반환하고 다음 라인으로 탈출)
    5.  `console.log("통신 발송 완료");` (서버에서 응답을 주기도 전에 이 동기 코드가 무조건 먼저 실행)
    6.  (잠시 후 서버 통신이 완료되면) -> `onreadystatechange` 콜백 내부의 `readyState 4` 동작이 최종 실행.
*   **결론**: 이처럼 요청을 보낸 뒤 결과를 기다리지 않고 다음 줄로 건너뛰어 다른 작업을 수행하다가, 나중에 통신 결과가 수령되면 등록된 리스너를 호출하는 흐름이 비동기식 처리 방식의 핵심입니다.

### 4) 실습 내용
*   XHR 통신 시 onreadystatechange 이벤트를 가로채어 readyState가 0부터 4까지 변화하는 전 과정을 터미널에 한 줄씩 로깅합니다.
*   통신 상태 완료(readyState === 4) 및 HTTP 성공 응답 코드(status === 200)를 판단하여 JSONPlaceholder의 유저 정보를 파싱해 화면에 카드로 렌더링합니다.

## 4. 콜백 패턴과 콜백 지옥(Callback Hell)

### 1) 콜백 지옥은 무한 루프인가?
*   **답은 아닙니다.** 콜백 지옥은 코드가 계속 반복 호출되어 무한히 도는 무한 루프 현상이 결코 아닙니다.
*   브라우저가 코드를 해석해 실행할 때는 1단계 -> 2단계 -> 3단계 순서대로 오차 없이 완벽하게 실행해 주며, 프로그램 구동 상에는 어떠한 오류도 없습니다.

### 2) 왜 지옥(Hell)이라고 부르는가?
컴퓨터가 아닌 **사람(개발자)이 코드를 분석하고 관리하기에 지옥 같다**는 의미에서 붙여진 이름입니다. 비동기 작업들을 순차 실행하기 위해 함수 내부에 또 함수를 넣고, 그 안에 또 함수를 중첩하는 형태로 작성하기 때문입니다.

```javascript
// 콜백 지옥의 실제 사례 (에러 처리가 결합된 중첩 구조)
fakeLogin("user@test.com", function(email, error) {
  if (error) {
    console.error("로그인 실패"); // 1단계 예외 처리
  } else {
    fakeGetProfile(email, function(profile, error) {
      if (error) {
        console.error("프로필 획득 실패"); // 2단계 예외 처리
      } else {
        fakeGetNotifications(profile, function(notifications, error) {
          if (error) {
            console.error("알림 조회 실패"); // 3단계 예외 처리
          } else {
            // 본래 하려던 최종 데이터 처리
            console.log("알림 목록 수신 완료:", notifications);
          }
        });
      }
    });
  }
});
```

*   **문제점 A: 가독성 파괴 (피라미드 소스코드)**
    *   비동기 작업이 3단계가 아니라 10단계, 15단계로 이어지면 코드가 오른쪽으로 계속 밀려나며 화면 밖으로 벗어납니다. 어떤 중괄호와 괄호가 어디서 닫히는지 확인조차 힘들어집니다.
*   **문제점 B: 예외 처리(에러 핸들링)의 누더기화**
    *   자바스크립트 표준 예외 처리 문법인 `try/catch` 문을 통틀어 단 한 번 사용할 수 없습니다. 비동기 함수의 안쪽에서 던져진 에러는 바깥의 `catch` 문으로 가닿지 못하기 때문에, 위 코드처럼 **매 단계마다 에러 분기(`if (error)`)를 중첩해서 따로 적어야만 합니다.** 만약 한 군데라도 분기를 빼먹으면 에러가 유실되어 어디서 통신이 터졌는지 디버깅이 불가능해집니다.

### 3) 실습 내용
*   모의 비동기 기능(setTimeout을 통한 비동기 시간 흐름)을 구현합니다.
*   [가상 로그인] -> [프로필 다운로드] -> [알림 수신] 순서대로 처리를 하기 위해 콜백 함수를 3중으로 꼬아 깊게 들어가는 코드를 실행하여, 콜백 지옥의 가독성 유실과 깊어지는 실행 로그의 흐름을 직접 체감합니다.
