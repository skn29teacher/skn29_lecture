# EC2 인스턴스 생성

1. 콘솔 검색 → **EC2** → **인스턴스 시작**
2. 이름: `skn29-django-server`
3. AMI(운영체제 이미지): **Ubuntu Server 26.04 LTS** (코드명: Resolute Raccoon, 2026년 4월 출시. Amazon Linux보다 Django 생태계 자료가 많아 추천)
   - EC2 콘솔의 AMI 선택 화면에서 **"Ubuntu"** 를 검색하면 최신 LTS가 상단에 표시됩니다
   - 소유자가 **Canonical** (공식 배포자)인지 확인하세요
4. 인스턴스 유형: **t3.micro** (프리티어 대상, t2.micro보다 성능이 좋고 최신 프리티어 기본 유형)
   >  리전에 따라 t2.micro만 프리티어인 경우도 있으니, 인스턴스 선택 화면에서 **"프리 티어 사용 가능"** 라벨을 확인하세요.
   >  **프리티어 정책이 변경되었습니다 (2025.07.15~)**: 2025년 7월 15일 이후에 생성한 AWS 계정은 기존의 서비스별 무료 한도 대신 **크레딧 기반 프리티어**가 적용됩니다. 가입 즉시 $100, 온보딩 과제 완료 시 추가 $100 (최대 $200)이 지급되며, 크레딧 소진 시 계정이 일시 중지됩니다 (기간: 6개월). 본인 계정의 프리티어 유형을 확인하려면 **AWS 콘솔 → Billing → Free Tier** 에서 확인하세요.
5. 키 페어(.pem) 생성
   - 키 페어 이름: `skn29-key`
   - 유형: RSA, 형식: `.pem`
   - **다운로드 후 반드시 안전한 곳에 보관** — 분실 시 서버 재접속 불가
   - 권장 저장 위치: `C:\Users\<사용자명>\.ssh\skn29-key.pem`
6. 네트워크 설정 → **기존 보안 그룹 선택** → 1차시에서 만든 `skn29-django-sg` 선택
7. 스토리지: 기본 8GB → 프리티어 한도(30GB gp3) 내에서 **20GB**로 변경
8. **인스턴스 시작** 클릭

>  인스턴스가 **"실행 중(Running)"** 상태가 될 때까지 1~2분 정도 소요됩니다. 상태 검사가 "2/2 통과"가 되면 접속 가능합니다.

### SSH 접속

#### 방법 A: MobaXterm으로 접속 (Windows 추천 — GUI + SFTP 통합)

[MobaXterm](https://mobaxterm.mobatek.net/)은 SSH 터미널, SFTP 파일 전송, X11 포워딩을 하나의 프로그램에서 지원하는 Windows 전용 원격 접속 도구입니다. PuTTY보다 편리하고 `.pem` 키를 변환 없이 바로 사용할 수 있어 AWS 실습에 적합합니다.

**1단계: MobaXterm 설치**

1. https://mobaxterm.mobatek.net/download.html 접속
2. **Home Edition** → **Portable edition** 다운로드 (설치 불필요, 압축 해제만으로 사용 가능)
   - 또는 **Installer edition** 선택 시 일반적인 설치 과정 진행
3. 압축 해제 후 `MobaXterm_Personal_xx.x.exe` 실행

**2단계: SSH 세션 생성**

1. 상단 메뉴에서 **Session** 클릭
2. 세션 유형에서 **SSH** 선택
3. 아래 항목을 입력:

| 항목 | 값 |
|------|-----|
| **Remote host** | EC2 퍼블릭 IPv4 주소 (예: `13.125.xxx.xxx`) |
| **Specify username** 체크 | `ubuntu` |
| **Port** | `22` |

4. **Advanced SSH settings** 탭 클릭
5. **Use private key** 체크 → 🔑 아이콘 클릭 → 다운로드한 `skn29-key.pem` 파일 선택
6. **OK** 클릭 → EC2 서버에 접속 완료!

**3단계: SFTP 파일 전송 (자동 지원)**

SSH 접속이 성공하면 MobaXterm 왼쪽에 **SFTP 브라우저**가 자동으로 열립니다.
- 로컬 PC의 파일을 드래그 & 드롭으로 서버에 업로드 가능
- 서버 파일을 로컬로 다운로드하는 것도 드래그 & 드롭으로 가능
- Django 프로젝트 파일을 업로드할 때 매우 편리합니다

>  **세션 저장**: MobaXterm은 한 번 만든 세션을 자동 저장합니다. 다음 접속 시에는 왼쪽 **Saved sessions**에서 더블클릭만 하면 바로 접속됩니다.

>  **멀티 탭 지원**: 여러 터미널 탭을 동시에 열어 한 탭에서는 서버 로그를 확인하고 다른 탭에서는 작업하는 식으로 활용 가능합니다.

**MobaXterm 접속 트러블슈팅**

| 증상 | 원인 | 해결 |
|------|------|------|
| "Connection refused" | 보안그룹에 22번 포트가 열리지 않음 | EC2 보안그룹 인바운드에 SSH(22) 추가 |
| "Network error: Connection timed out" | IP 주소가 잘못되었거나 인스턴스가 꺼져 있음 | EC2 콘솔에서 퍼블릭 IP 재확인 & 인스턴스 상태 확인 |
| "Server refused our key" | 사용자 이름이 잘못됨 (예: `root` 대신 `ubuntu`) | Username을 `ubuntu`로 수정 |
| .pem 파일을 찾을 수 없음 | 파일 선택 대화상자에서 필터가 `*.ppk`로 되어 있음 | 파일 형식 필터를 **All files (\*.\*)** 로 변경 |

#### 방법 B: PowerShell / Windows Terminal로 접속

```powershell
# pem 파일 권한 설정 (최초 1회)
icacls.exe skn29-key.pem /reset
icacls.exe skn29-key.pem /grant:r "$($env:USERNAME):(R)"
icacls.exe skn29-key.pem /inheritance:r

# SSH 접속
ssh -i "skn29-key.pem" ubuntu@<퍼블릭 IP 주소>
```

>  PowerShell에서 `WARNING: UNPROTECTED PRIVATE KEY FILE!` 오류가 나면 위의 `icacls` 명령을 정확히 실행했는지 확인하세요. 키 파일에 다른 사용자의 읽기 권한이 남아 있으면 SSH가 거부합니다.

#### 방법 C: Mac/Linux 터미널

```bash
chmod 400 skn29-key.pem
ssh -i "skn29-key.pem" ubuntu@<퍼블릭 IP 주소>
```

>  퍼블릭 IP는 EC2 콘솔 → 인스턴스 클릭 → **"퍼블릭 IPv4 주소"**에서 확인. 인스턴스를 중지 후 재시작하면 IP가 바뀔 수 있으니(탄력적 IP 미사용 시) 매번 확인 필요.

>  **처음 접속 시** "Are you sure you want to continue connecting (yes/no)?" 메시지가 뜨면 `yes`를 입력하세요. 이는 서버의 호스트 키를 로컬에 저장하는 정상적인 과정입니다.

# 보안 그룹이란?

- EC2 인스턴스 단위로 적용되는 **가상 방화벽**
- 허용할 IP 대역과 포트를 정의하는 화이트리스트(Whitelist) 방식의 통신 통제 도구
- **기본 상태**: 모든 인바운드(들어오는) 트래픽은 차단, 아웃바운드(나가는) 트래픽은 전체 허용

### 인바운드 vs 아웃바운드

| 구분 | 흐름 | 예시 |
|------|------|------|
| **Inbound (인바운드)** | 외부 → 내 EC2 서버 | 사용자가 내 웹 서비스 주소(80, 443포트)로 접근하는 흐름 |
| **Outbound (아웃바운드)** | 내 EC2 서버 → 외부 | 내 서버 프로그램이 외부 AI API(Anthropic, OpenAI 등)를 호출하는 흐름 |

### Django 서버용 보안 그룹 선제 생성

인스턴스 생성 전 보안 그룹을 미리 만들어 설계 규칙을 적용합니다.

1. 콘솔 검색창에 **"EC2"** 입력 → 좌측 메뉴 **Network & Security (네트워크 및 보안) → Security Groups (보안 그룹)**
2. **Create security group (보안 그룹 생성)** 클릭
3. 기본 세부 정보 입력:
   - Security group name: `skn29-django-sg`
   - Description: `Django web server security group - SSH/HTTP/HTTPS/8000`
   - VPC: 기본(Default) VPC 선택 확인
4. **Inbound rules (인바운드 규칙)** 추가 (아래 설계표 규칙 입력)
5. **Outbound rules (아웃바운드 규칙)**: 기본값(All traffic 허용) 그대로 유지
6. **Create security group** 클릭

#### 인바운드 규칙 설계표

| 유형 (Type) | 프로토콜 | 포트 범위 | 소스 (Source) | 설명 (Description) |
|------|----------|-----------|------|------|
| **SSH** | TCP | 22 | **내 IP (My IP)** | 서버 원격 접속 포트 (보안을 위해 절대 `0.0.0.0/0` 금지) |
| **HTTP** | TCP | 80 | **Anywhere-IPv4 (0.0.0.0/0)** | 일반 웹 서비스 포트 |
| **HTTPS** | TCP | 443 | **Anywhere-IPv4 (0.0.0.0/0)** | 보안 웹 서비스 포트 (SSL/TLS 적용 후) |
| **사용자 지정 TCP** | TCP | 8000 | **내 IP (My IP)** | Django `runserver` 개발 서버 접근용 포트 |

**보안 수칙 및 팁**:
> - **SSH(22번 포트)**와 **개발 포트(8000번 포트)**는 반드시 **내 IP**로 지정하세요. 전 세계 모든 IP(`0.0.0.0/0`)에 개방하면 무차별 대입 공격(Brute Force)의 표적이 됩니다.
> - 카페, 학교, 집 등 접속 장소를 변경하면 로컬 컴퓨터의 퍼블릭 IP가 달라집니다. 이때 EC2 접속이 안 되는 현상이 발생하므로, 장소를 옮겼을 때 접속이 차단되면 **보안 그룹 설정으로 와서 인바운드 규칙의 소스를 다시 "내 IP"로 갱신**해 주어야 합니다.

# Gunicorn 단독 테스트 실행(정상동작 확인)
gunicorn --bind 0.0.0.0:8000 config.wsgi:application

# 서버가 재 부팅될때 자동으로 Gunicorn을 실행하도록 systemd 등록
sudo vim /etc/systemd/system/gunicorn.service
```
[Unit]
Description=gunicorn daemon for Django
After=network.target

[Service]
User=ubuntu
Group=www-data
WorkingDirectory=/home/ubuntu/myproject
ExecStart=/home/ubuntu/myproject/venv/bin/gunicorn \
          --workers 3 \
          --bind unix:/home/ubuntu/myproject/gunicorn.sock \
          config.wsgi:application

[Install]
WantedBy=multi-user.target
```

# nginx 사이트 설정 파일 생성
sudo vim /etc/nginx/sites-available/myproject
```
server {
    listen 80;
    server_name <EC2 퍼블릭 IP>;

    location = /favicon.ico { access_log off; log_not_found off; }

    location /static/ {
        alias /home/ubuntu/myproject/static/;
    }

    location / {
        include proxy_params;
        proxy_pass http://unix:/home/ubuntu/myproject/gunicorn.sock;
    }
}
```

# 트러블슈팅


| 증상 | 원인 후보 | 점검 명령어 |
|------|-----------|--------------| 
| 502 Bad Gateway | Gunicorn이 안 떠 있음 / 소켓 파일 없음 | `sudo systemctl status gunicorn` → `sudo journalctl -u gunicorn` |
| 502 Bad Gateway | 소켓 파일 경로 불일치 | Nginx 설정의 `proxy_pass` 경로와 gunicorn.service의 `--bind` 경로가 같은지 확인 |
| 403 Forbidden | Nginx가 소켓 파일에 접근 불가 | `chmod 755 /home/ubuntu` 실행, `ls -la /home/ubuntu/gunicorn.sock` 확인 |
| 정적파일 깨짐(CSS 미적용) | `collectstatic` 미실행 | `python manage.py collectstatic --noinput` |
| 정적파일 404 | Nginx에서 `root` vs `alias` 혼동 | Nginx 설정에서 `alias /home/ubuntu/myproject/static/;` 확인 |
| 변경사항 반영 안됨 | 서비스 재시작 누락 | `sudo systemctl restart gunicorn nginx` |
