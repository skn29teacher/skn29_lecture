# 7차시: 프로세스 관리 및 포트 충돌 장애 대응

## 1. 학습 목표
* 현재 리눅스 CPU와 메모리에서 실행 중인 모든 프로세스 상태를 실시간 진단합니다.
* 포트 충돌이나 좀비 프로세스로 장애가 발생했을 때, 원인 프로세스를 찾아 강제 종료하는 방법을 익힙니다.
* 포그라운드/백그라운드 작업 전환을 이해하고, 실무 배포 시 발생하는 프로세스 문제를 스스로 해결합니다.

---

## 2. 관련 이론

### 2-1. 프로세스란?

프로세스는 실행 중인 프로그램의 인스턴스입니다. 같은 프로그램이라도 여러 번 실행하면 각각 다른 PID(Process ID)를 받아 독립적으로 동작합니다.

| 용어 | 설명 |
|------|------|
| PID | 운영체제가 각 프로세스에 부여한 고유 번호 |
| PPID | 부모 프로세스 ID (해당 프로세스를 실행시킨 프로세스) |
| 포그라운드 | 현재 터미널을 점유하며 실행 중인 프로세스 |
| 백그라운드 | 터미널과 분리되어 뒤에서 실행되는 프로세스 |
| 데몬(Daemon) | 시스템 부팅 시 시작해 항상 백그라운드에서 돌아가는 서비스 프로세스 |
| 좀비 프로세스 | 실행이 끝났지만 부모 프로세스가 회수하지 않아 프로세스 테이블에 남은 상태 |

### 2-2. ps 명령어 출력 항목 해석

`ps aux` 실행 시 출력 컬럼 의미:

| 컬럼 | 의미 |
|------|------|
| USER | 프로세스 소유 계정 |
| PID | 프로세스 ID |
| %CPU | CPU 사용률 |
| %MEM | 메모리 사용률 |
| VSZ | 가상 메모리 크기 (KB) |
| RSS | 실제 사용 중인 물리 메모리 (KB) |
| STAT | 프로세스 상태 (R: 실행중, S: 슬립, Z: 좀비, D: I/O 대기) |
| START | 프로세스 시작 시간 |
| TIME | 누적 CPU 사용 시간 |
| COMMAND | 실행 명령어 |

### 2-3. 포트 충돌 원리

```
[포트 8080]
│
├── 프로세스 A (PID: 1234) - 이미 점유 중
└── 프로세스 B (새로 실행 시도) → "Address already in use" 오류!
```

하나의 포트에는 동시에 단 하나의 프로세스만 바인딩될 수 있습니다. 이미 점유 중인데 새 서비스가 같은 포트로 실행되면 즉시 오류가 발생합니다.

**자주 쓰이는 기본 포트**:

| 포트 | 서비스 |
|------|--------|
| 22 | SSH |
| 80 | HTTP |
| 443 | HTTPS |
| 3000 | Node.js 개발 서버 |
| 5000 | Flask 기본 포트 |
| 8080 | Spring Boot 기본 포트 |
| 5432 | PostgreSQL |
| 3306 | MySQL |
| 6379 | Redis |

### 2-4. kill 시그널 종류

| 시그널 | 번호 | 의미 | 사용 시점 |
|--------|------|------|----------|
| SIGTERM | 15 | 정상 종료 요청 (기본값) | 일반적인 종료 (`kill PID`) |
| SIGKILL | 9 | 강제 즉시 종료 | 응답 없는 프로세스 (`kill -9 PID`) |
| SIGHUP | 1 | 설정 재로드 | nginx, sshd 등 데몬 설정 갱신 |
| SIGINT | 2 | 키보드 인터럽트 | `Ctrl+C`와 동일 |

> **SIGTERM vs SIGKILL**: SIGTERM(15)을 받은 프로세스는 정리 작업(파일 닫기, DB 커넥션 반납 등)을 마치고 종료합니다. SIGKILL(9)은 OS가 즉시 강제 종료하므로 정리 작업 없이 죽습니다. 먼저 15로 시도하고, 응답 없으면 9를 씁니다.

### 2-5. 포그라운드/백그라운드 전환

```bash
명령어 &        # 처음부터 백그라운드로 실행
Ctrl + Z       # 실행 중인 포그라운드 프로세스를 일시 정지 후 백그라운드로 전환
jobs           # 현재 터미널의 백그라운드 작업 목록
fg %번호       # 백그라운드 작업을 포그라운드로 복귀
bg %번호       # 일시 정지된 작업을 백그라운드에서 계속 실행
```

## 3. 실습 단계 및 명령어

### 실습 1: 전체 프로세스 조회
```bash
ps aux

# 출력이 많을 때 특정 프로세스만 필터링
ps aux | grep python
ps aux | grep nginx

# 프로세스를 CPU 사용량 순으로 정렬
ps aux --sort=-%cpu | head -20

# 메모리 사용량 순으로 정렬
ps aux --sort=-%mem | head -20
```

### 실습 2: 특정 포트 점유 프로세스 조회
```bash
# 포트 8080을 점유한 프로세스 확인
lsof -i :8080

# 여러 포트 동시 확인
lsof -i :8080 -i :3000 -i :5000

# 모든 LISTEN 상태 포트 확인
lsof -i -P -n | grep LISTEN

# netstat으로 확인 (net-tools 패키지 필요)
netstat -tulpn | grep 8080
```

### 실습 3: 백그라운드 작업 기동 및 전환
```bash
# 백그라운드로 실행
sleep 300 &
sleep 400 &

# 현재 백그라운드 작업 목록
jobs
# [1]+  Running    sleep 300 &
# [2]-  Running    sleep 400 &

# 1번 작업을 포그라운드로 복귀
fg %1
# (복귀 후 Ctrl+C 로 정지)

# 포그라운드 프로세스를 백그라운드로 전환
sleep 500
# Ctrl+Z 로 일시 정지
bg %1   # 백그라운드에서 계속 실행
```

### 실습 4: 프로세스 강제 종료
```bash
# 백그라운드 프로세스 하나 실행
sleep 1000 &

# PID 확인
ps aux | grep sleep

# PID가 예를 들어 234라면:
# 정상 종료 시도 (SIGTERM)
kill 234

# 종료 안 될 경우 강제 종료 (SIGKILL)
kill -9 234

# 프로세스 이름으로 한 번에 종료
killall sleep
pkill -f "sleep 1000"
```

### 실습 5: 실시간 프로세스 모니터링
```bash
# htop 설치 후 실행 (인터랙티브 모니터)
apt install -y htop
htop
# F10 또는 q로 종료

# top 명령어 (기본 내장)
top
# q로 종료, P: CPU순 정렬, M: 메모리순 정렬
```

## 4. 포트 충돌 장애 대응 플로우

실제 운영에서 `Address already in use` 오류 발생 시 처리 순서:

```
1. 오류 확인
   → "Port 8080 is already in use"

2. 점유 프로세스 탐색
   → lsof -i :8080

3. PID 확인 후 프로세스 정보 파악
   → ps aux | grep [PID]
   → 어떤 서비스인지, 종료해도 되는지 확인

4. 정상 종료 시도
   → kill [PID]

5. 응답 없으면 강제 종료
   → kill -9 [PID]

6. 포트 해제 확인 후 서비스 재시작
   → lsof -i :8080  (아무것도 안 나와야 정상)
```

## 5. 서버 운영 관점의 실전 시나리오

### 시나리오 A: 포트 충돌 장애 즉시 해결
```bash
# 스프링 부트 재배포 중 오류 발생
# Error: Web server failed to start. Port 8080 was already in use.

# 1. 점유 프로세스 확인
lsof -i :8080
# COMMAND   PID USER   TYPE
# java     1234 root   IPv4  LISTEN

# 2. 어떤 프로세스인지 확인
ps aux | grep 1234

# 3. 이전 배포 잔존 프로세스 제거
kill -9 1234

# 4. 포트 해제 확인
lsof -i :8080   # 아무것도 안 나와야 정상

# 5. 신규 배포 재실행
./deploy.sh
```

### 시나리오 B: 백그라운드로 서버 실행
```bash
# 개발 서버를 백그라운드에서 돌리기
python3 app.py &
echo "서버 PID: $!"  # $!는 가장 최근 백그라운드 프로세스 PID

# PID를 파일에 저장해두면 나중에 종료할 때 편리
python3 app.py &
echo $! > /tmp/app.pid

# 나중에 종료 시
kill $(cat /tmp/app.pid)
```

### 시나리오 C: CPU/메모리 과부하 프로세스 추적
```bash
# CPU 90% 이상 사용 중인 프로세스 찾기
ps aux --sort=-%cpu | head -5

# 메모리 500MB 이상 사용 프로세스 찾기
ps aux --sort=-%mem | awk '$6 > 500000 {print}'

# 특정 프로세스의 리소스 실시간 추적
top -p [PID]
```
